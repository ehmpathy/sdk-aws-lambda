dream = migrate error types: BadRequestError -> ConstraintError, UnexpectedCodePathError -> MalfunctionError

## .why

align with helpful-errors semantics:

| current | new | exit code | semantics |
|---------|-----|-----------|---------|
| BadRequestError | ConstraintError | 2 | caller must fix |
| UnexpectedCodePathError | MalfunctionError | 1 | server must fix |

## .what

replace throughout sdk-aws-lambda:
- `BadRequestError` -> `ConstraintError`
- `UnexpectedCodePathError` -> `MalfunctionError`

ConstraintError and MalfunctionError already in `helpful-errors`

## .scope

- handler error responses: `errorType: 'BadRequestError'` -> `errorType: 'ConstraintError'`
- client error detection: detect `errorType === 'ConstraintError'`
- all throws of BadRequestError/UnexpectedCodePathError

## .consider

- backwards compat: callers may check for `BadRequestError` by name
- ConstraintError/MalfunctionError already exported by helpful-errors
- exit code semantics: ConstraintError.exit = 2, MalfunctionError.exit = 1

## .not now

defer to later: focus on current behavior validation first
