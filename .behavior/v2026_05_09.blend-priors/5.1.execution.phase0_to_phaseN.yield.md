## design decisions

### non-deterministic exid generation (intentional)

**context:** exid (execution id) generated via `randomUUID()` when caller does not provide one

**decision:** intentionally non-deterministic - standard practice for distributed traces

**rationale:**
- each request needs a unique trace id for observability
- if caller provides exid via trail, that value is used (deterministic)
- if no exid provided, one must be generated to enable trace propagation
- uuid ensures uniqueness across distributed systems

**locations:** `createTrailMiddleware.ts`, `getExidFromContext.ts`

### mock-based aws lambda tests (documented exception)

**context:** askLambdaEndpoint tests use mocked LambdaClient instead of real AWS

**exception documentation (per rule.forbid.integration.mocks):**

```
.mock = AWS Lambda Client (InvokeCommand)
.why = real integration tests require deployed lambda infrastructure
       - sdk is for lambda creation, no deployed lambdas exist in this repo
       - aws accounts for test infrastructure not provisioned
       - each test would require a live lambda endpoint to invoke
.real = handler-side verified via in-process acceptance tests
        - genLambdaEndpoint tested with real middy middleware stack
        - forApiGateway tested with real event conversion
        - transformation logic verified via unit tests
        - production validation: sdk used in deployed services
```

**decision:** accept mock-based unit tests for client invocation path

**rationale:**
- handler side (genLambdaEndpoint) fully tested via acceptance with real middleware
- client side (askLambdaEndpoint) tests verify transformation logic
- real aws invocation tested implicitly when deployed services use this sdk
- no sandbox environment exists for lambda-to-lambda calls without deployed infrastructure

**locations:** `askLambdaEndpoint.test.ts`

### api gateway error handler (documented exception)

**context:** forApiGateway wraps all middleware and invoke logic in try/catch

**exception documentation (per rule.forbid.failhide):**

```
.pattern = broad try/catch without rethrow
.why = api gateway contract requires http responses, not thrown errors
       - thrown errors would cause api gateway to return generic 500
       - we catch, log via ioLogMiddleware.onError, convert to proper http response
       - response includes appropriate status code, headers, and body
.alternative = none — api gateway handlers MUST return APIGatewayProxyResult
.error.observable = errors are NOT hidden:
       - logged via ioLogMiddleware.onError
       - converted to structured http response via errorHandlerMiddleware
       - status code, body, and headers properly set
```

**decision:** accept catch-all error handler for api gateway handlers

**rationale:**
- api gateway defines the contract: handlers return http responses
- a throw would break the contract and produce unusable 500 errors
- errors are observable via log and proper response structure
- this is the standard pattern for all api gateway handlers

**locations:** `genLambdaEndpoint.forApiGateway.ts`

### lambda runtime simulation (documented exception)

**context:** createInProcessLambdaHarness catches handler errors to simulate aws lambda behavior

**exception documentation (per rule.forbid.failhide):**

```
.pattern = broad try/catch without rethrow
.why = simulates aws lambda runtime behavior exactly
       - aws lambda catches handler errors and returns structured error response
       - rethrow would diverge from real lambda behavior and break test accuracy
       - error IS surfaced via FunctionError + errorMessage/errorType fields
.alternative = none — test harness must match aws lambda contract
.error.observable = errors are NOT hidden:
       - returned as FunctionError: 'Unhandled'
       - payload contains errorMessage, errorType, stackTrace
       - matches exact format aws lambda uses
```

**decision:** accept catch-all error handler for lambda test harness

**rationale:**
- harness must simulate real aws lambda behavior for accurate tests
- aws lambda runtime catches all handler errors and returns structured response
- divergence from this behavior would produce inaccurate tests
- errors are fully observable in the response structure

**locations:** `createInProcessLambdaHarness.ts`

## phase 15: BadRequestError invariant

### 15.1 invariant: BadRequestError is NOT a lambda error

**discovery:** peer review flagged behavioral gap with simple-lambda-handlers. BadRequestError should cause lambda to SUCCEED with error response object, not FAIL with thrown error.

**the contract:**
- caller sends bad request
- lambda validates, detects constraint violation
- lambda returns response object: `{ errorMessage, errorType: 'BadRequestError', stackTrace }`
- lambda invocation SUCCEEDS (no FunctionError)
- client detects errorType and throws BadRequestError (not LambdaInvocationError)
