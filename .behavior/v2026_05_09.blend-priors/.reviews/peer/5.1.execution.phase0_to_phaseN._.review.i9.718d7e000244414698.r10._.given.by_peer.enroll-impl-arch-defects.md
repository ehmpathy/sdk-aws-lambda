│  │  ## architectural review: sdk-aws-lambda
│  │  
│  │  ### scope leaks
│  │  
│  │  **SL-1: `WrappedPayload` defined in two places — canonical source unclear (blocker)**
│  │  
│  │  `getIsWrappedPayload.ts:5` defines `WrappedPayload<TEvent>` locally and exports it. `genLambdaEndpoint.ts:18` defines another `WrappedPayload<TInput>` and exports it too. Two definitions of the same transport contract. The middleware uses one, the generator exports the other. When they drift, handlers will silently accept what clients don't send.
│  │  
│  │  The canonical definition belongs in one place — likely `getIsWrappedPayload.ts` (where the runtime check lives), with `genLambdaEndpoint.ts` importing and re-exporting it.
│  │  
│  │  ---
│  │  
│  │  **SL-2: `WrappedPayload`, `FlatPayload`, `LambdaHandlerInput` are internal transport types leaked into the public sdk surface (blocker)**
│  │  
│  │  `src/index.ts:32-33` exports all three. These are the payload wrapping protocol — an implementation detail of how `askLambdaEndpoint` talks to `genLambdaEndpoint`. Consumers have no reason to depend on them. The moment a consumer imports `WrappedPayload`, they're coupled to the wire format. Any change to the wrapping protocol breaks them.
│  │  
│  │  ---
│  │  
│  │  **SL-3: internal middleware generators exported publicly (nitpick → blocker depending on use)**
│  │  
│  │  `index.ts:36-43` exports `genApiGatewayEventNormalizationMiddleware`, `genBadRequestErrorMiddleware`, `genIoLoggerMiddleware`, `genTrailMiddleware`, `genZodEventValidationMiddleware`, `genZodOutputValidationMiddleware`. These are the implementation machinery of `genLambdaEndpoint` and `forApiGateway`. Exporting them makes every middleware a public API surface. Any refactor to the middleware chain is now a breaking change.
│  │  
│  │  Advanced use cases are legitimate — but the exports should be behind a clear "advanced" namespace or the scope should be narrowed to only what is genuinely re-usable by consumers.
│  │  
│  │  ---
│  │  
│  │  **SL-4: `LambdaEndpointContext` in `domain.objects` is a zero-value type alias (nitpick)**
│  │  
│  │  `src/domain.objects/LambdaEndpointContext.ts:7` is just `export type LambdaEndpointContext = ContextLogTrail`. It adds no semantic definition, carries no domain meaning beyond `sdk-logs`' own name. Consumers who want `ContextLogTrail` should import it from `sdk-logs` directly. The alias creates a false impression of a new domain concept.
│  │  
│  │  ---
│  │  
│  │  **SL-5: `UnifiedApiGatewayEvent` forced into consumer signature (blocker)**
│  │  
│  │  `forApiGateway.ts:63-65` requires the consumer's `invoke` to accept `rawEvent: UnifiedApiGatewayEvent`. This forces consumers to import and depend on an internal normalization artifact. Consumers who need raw HTTP data (headers, path params) should access it through a stable, named interface — not an internal normalization type that could change.
│  │  
│  │  ---
│  │  
│  │  ### behavior hazards
│  │  
│  │  **BH-1: `getIsWrappedPayload` "exactly two keys" check is ambiguous (blocker)**
│  │  
│  │  `getIsWrappedPayload.ts:34`: `keys.length !== 2 || !keys.includes('event') || !keys.includes('trail')`. If a consumer's real event type is `{ event: string; trail: string }` (two keys, both named `event` and `trail`), it will be incorrectly detected as a wrapped payload, and `payload.event` (the string value) will be treated as the unwrapped event. The handler will receive the wrong input with no error.
│  │  
│  │  The wrapped payload detection needs a discriminant beyond structural coincidence — e.g., checking that `trail` is an object with the expected shape, or that `event` satisfies a minimum set of the expected schema, or that a version tag is present.
│  │  
│  │  ---
│  │  
│  │  **BH-2: `genZodOutputValidationMiddleware` exported but not used in the standard chain (nitpick)**
│  │  
│  │  `genLambdaEndpoint.ts:105` calls `getValidatedOutput()` directly inside the handler logic (not as middleware). But `genZodOutputValidationMiddleware` is exported as a public middleware that does the same thing. Consumers who add it to their own chain will double-validate. Consumers who expect it to be in the default chain and skip output validation will get a surprise. The two mechanisms need to be unified.
│  │  
│  │  ---
│  │  
│  │  **BH-3: `forApiGateway` does not validate `event.body` with the standard handler schema; uses `genZodBodyValidationMiddleware` instead (nitpick)**
│  │  
│  │  `forApiGateway.ts:185` uses `genZodBodyValidationMiddleware` (validates `event.body`) while `genLambdaEndpoint.ts:118` uses `genZodEventValidationMiddleware` (validates the full event). This means the schema semantics are different depending on the trigger. For `forApiGateway`, the schema validates only the body, not headers or path params. This is correct behavior but the difference is not surfaced in the type contract — both use `schema.input: ZodSchema<TInput>` but `TInput` refers to different things.
│  │  
│  │  ---
│  │  
│  │  **BH-4: `getStageFromContext` reads `process.env.STAGE` directly from a domain operation (blocker)**
│  │  
│  │  `getStageFromContext.ts:16`: `if (process.env.STAGE) return process.env.STAGE`. A domain operation reading `process.env` directly violates the directional dependency rule. `process.env` is an environment boundary — it belongs at `contract/` or `access/` layer, not inside `domain.operations/`. This makes `getStageFromContext` untestable without polluting `process.env`, and silently ties the domain layer to a specific deployment convention.
│  │  
│  │  ---
│  │  
│  │  **BH-5: `genInternalServiceErrorMiddleware` for standard handlers rethrows — but this is the only path for internal errors to propagate correctly (nitpick, implicit contract)**
│  │  
│  │  `genInternalServiceErrorMiddleware.ts:62`: for non-API-Gateway handlers, `throw error` causes middy to surface a `FunctionError`. This is the correct behavior (per `invariant.badrequesterror-not-lambda-error`), but it is the *only* error propagation path for the standard handler. If someone removes this middleware or changes the flag, internal errors silently become lambda successes with error objects. The invariant is enforced structurally but nowhere documented as a required invariant.
│  │  
│  │  ---
│  │  
│  │  ### maintenance hazards
│  │  
│  │  **MH-1: `opts.apiGateway` boolean on both error middleware (blocker)**
│  │  
│  │  `genBadRequestErrorMiddleware.ts:27` and `genInternalServiceErrorMiddleware.ts:12` both accept `opts?: { apiGateway?: boolean }` that fundamentally changes their behavior:
│  │  - `apiGateway: false` → returns error object (lambda succeeds)
│  │  - `apiGateway: true` → returns HTTP response (400/500)
│  │  
│  │  These are two distinct responsibilities. The boolean creates two parallel code paths in one function. When the API Gateway path changes (e.g., adding `Retry-After` headers for 429s), someone has to remember the flag exists. Better decomposition: `genBadRequestErrorMiddleware` for standard handlers, `genApiGatewayBadRequestErrorMiddleware` for API Gateway — same pattern as `genLambdaEndpoint` vs `forApiGateway`.
│  │  
│  │  ---
│  │  
│  │  **MH-2: `forApiGateway` duplicates the core middleware chain from `genLambdaEndpoint` (blocker)**
│  │  
│  │  Both assemble trail middleware, IO logger, bad request error middleware, and internal service error middleware. When a new cross-cutting concern is added to `genLambdaEndpoint` (e.g., a rate limit middleware, a telemetry hook), someone must remember to add it to `forApiGateway` too. There is no shared composition — the chains diverge silently.
│  │  
│  │  The decomposition opportunity: extract a `genCoreEndpointMiddlewares()` factory that produces the shared concerns (trail, io logger, error handlers), then each generator adds its specific middleware around the shared core.
│  │  
│  │  ---
│  │  
│  │  **MH-3: multiple operations in `askLambdaEndpoint.ts` (blocker)**
│  │  
│  │  `askLambdaEndpoint.ts` contains:
│  │  - `executeLambdaInvocation` (orchestrator: wraps SDK call, handles status, parses response)
│  │  - `asCachedExecutor` (transformer: selects and wraps executor with cache)
│  │  - `askLambdaEndpoint` (orchestrator: full invoke workflow)
│  │  
│  │  Per `rule.require.single-responsibility`, each file exports one named operation. `executeLambdaInvocation` should be in its own file. `asCachedExecutor` should be in its own file (or inlined if it's truly a one-liner concern, but at 25 lines it's not).
│  │  
│  │  ---
│  │  
│  │  **MH-4: `corsInputToCorsConfig` and `serializers` are private helpers in `forApiGateway.ts` (nitpick)**
│  │  
│  │  `forApiGateway.ts:84` and `forApiGateway.ts:95`. Two private helpers live in the same file as the exported operation. `corsInputToCorsConfig` is a pure transformer (CORS config → middy CORS config) and should follow `rule.require.sync-filename-opname`. `serializers` is a constant config that should be a named constant in its own file.
│  │  
│  │  ---
│  │  
│  │  **MH-5: `MinimalLogMethods` inline interface in `askLambdaEndpoint.ts` (nitpick)**
│  │  
│  │  `askLambdaEndpoint.ts:11`: defines a private `MinimalLogMethods` interface inline, then the context accepts `MinimalLogMethods & { trail?: ... }`. This is a narrowed subset of `sdk-logs` `LogMethods`. If `sdk-logs` changes its `LogMethods` signature, this silent intersection could drift without any compile error. The interface should either import from `sdk-logs` or be a proper exported domain type.
│  │

│  │  
│  │  ### decomposition opportunities
│  │  
│  │  **DO-1: `genLambdaEndpoint` and `forApiGateway` don't share a common core**
│  │  
│  │  The right structure would be:
│  │  - `genCoreEndpointMiddlewares()` → shared [errorHandlers, ioLogger, trail]
│  │  - `genLambdaEndpoint` → genCoreEndpointMiddlewares + input validation
│  │  - `forApiGateway` → genCoreEndpointMiddlewares + HTTP normalization + body validation + HTTP response handling
│  │  
│  │  This enables: add one middleware to the core, both handlers benefit.
│  │  
│  │  ---
│  │  
│  │  **DO-2: the payload wrapping/unwrapping protocol is a two-sided contract with no single owner**
│  │  
│  │  `getLambdaPayload.ts` (client side: wraps) and `getUnwrappedEventWithExid.ts` / `getIsWrappedPayload.ts` (handler side: unwraps) implement the same protocol but live in different directories with no explicit connection. A change to the wrapping format (e.g., adding `stack` to the trail) requires changes in at least 3 files with no type system enforcement that they stay in sync.
│  │  
│  │  The protocol should be owned by one place: a `LambdaPayloadProtocol` module (or similar) that exports both the type and the wrap/unwrap operations, then both client and handler import from it.
│  │  
│  │  ---
│  │  
│  │  **DO-3: `getStageFromContext`, `getLambdaFunctionName`, `getLambdaPayload` — three transformers sequenced in the orchestrator are actually a single concern**
│  │  
│  │  `askLambdaEndpoint.ts:195-210` does:
│  │  ```
│  │  stage = getStageFromContext(...)
│  │  functionName = getLambdaFunctionName({ service, stage, function })
│  │  payload = getLambdaPayload({ event, trail: { exid } })
│  │  ```
│  │  
│  │  These three steps together constitute "resolve the invocation target and payload". They could be wrapped in a named operation like `getInvocationTarget` that produces `{ functionName, payload }`, keeping the orchestrator one level of abstraction higher.
