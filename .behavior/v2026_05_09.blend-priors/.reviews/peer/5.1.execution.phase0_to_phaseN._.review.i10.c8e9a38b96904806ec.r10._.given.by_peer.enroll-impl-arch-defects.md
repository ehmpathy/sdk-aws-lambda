│  │  ## architectural review: sdk-aws-lambda
│  │  
│  │  ### behavior hazards
│  │  
│  │  **[B1] input never logged — ioLogger runs before trail injects log** ← blocker
│  │  
│  │  in both `genLambdaEndpoint` and `forApiGateway`, `genIoLoggerMiddleware` appears at index 2 and `genTrailMiddleware` at index 3 (or later). middy runs `before` handlers in array order. at `ioLogger.before` time, `context.log` is `undefined` — trail hasn't injected it yet. the `if (log)` guard silently skips. **inputs are never logged.**
│  │  
│  │  ```
│  │  before chain (as ordered):
│  │    2: ioLogger.before   ← context.log = undefined → skips silently
│  │    3: trail.before      ← injects context.log
│  │    4: validation.before
│  │  ```
│  │  
│  │  after chain runs in reverse, so `ioLogger.after` runs with log available — outputs are logged. only inputs are broken.
│  │  
│  │  fix: move `genTrailMiddleware()` before `genIoLoggerMiddleware()` in both chains.
│  │  
│  │  ---
│  │  
│  │  **[B2] `getIsWrappedPayload` — false positive on real events with `{event, trail}` keys** ← blocker
│  │  
│  │  ```ts
│  │  if (keys.length !== 2 || !keys.includes('event') || !keys.includes('trail'))
│  │    return false;
│  │  ```
│  │  
│  │  any user event whose schema happens to have exactly two keys named `event` and `trail` will be misidentified as a wrapped payload, silently unwrapped, and the handler will receive `event.event` (wrong). no error. no logging. pure silent misbehavior.
│  │  
│  │  also a forward-compat hazard: if a third key is ever added to the wrapped payload (e.g., `version`), detection silently breaks in the other direction.
│  │  
│  │  the two-key exact match is too fragile as a discriminant. options:
│  │  - use a versioned discriminant field in the wrapper (`__sdk: 'sdk-aws-lambda/v1'`)
│  │  - or use a more resilient heuristic (e.g., only detect wrapped if `trail` has `exid` as string or undefined and `event` is a non-null object)
│  │  
│  │  ---
│  │  
│  │  **[B3] `UnifiedApiGatewayEvent.trail` is defined but never extracted** ← blocker (dead field misleads)
│  │  
│  │  ```ts
│  │  export interface UnifiedApiGatewayEvent {
│  │    ...
│  │    trail?: { exid?: string };   // ← declared but never read
│  │  }
│  │  ```
│  │  
│  │  `genTrailMiddleware` calls `getIsWrappedPayload` on the normalized event. a `UnifiedApiGatewayEvent` has 7+ keys, never passes the 2-key check. trail is never extracted. the `trail` field on `UnifiedApiGatewayEvent` is dead API surface that implies trail propagation works through API Gateway — it doesn't.
│  │  
│  │  either remove the field, or implement extraction from API Gateway headers/body (the right design).
│  │  
│  │  ---
│  │  
│  │  **[B4] `genApiGatewayEventNormalizationMiddleware` — silent pass-through on unknown event** ← blocker
│  │  
│  │  ```ts
│  │  if (getIsV1ApiGatewayEvent(event)) { ... }
│  │  if (getIsV2ApiGatewayEvent(event)) { ... }
│  │  // else: nothing. unknown event silently passes through.
│  │  ```
│  │  
│  │  if neither v1 nor v2 event is detected, the raw event passes through unchanged. downstream body validation will fail with an opaque type error instead of a clear "unrecognized event format" message. should throw an `UnexpectedCodePathError` in the `else` branch.
│  │  
│  │  ---
│  │  
│  │  ### maintenance hazards
│  │  
│  │  **[M1] `genLambdaEndpoint` and `forApiGateway` are parallel factories with no shared core** ← blocker
│  │  
│  │  both independently assemble the error/trail/logging middleware chain. any change to the core ordering or addition of a new core middleware must be made in two places. `forApiGateway` was intended as an adapter on top of `genLambdaEndpoint` (for API Gateway specifics) but ended up a full copy.
│  │  
│  │  decomp opportunity: extract a `genLambdaEndpointCore()` that assembles the shared chain (bad request, internal service, io logger, trail, validation), then both can compose from it.
│  │  
│  │  ---
│  │  
│  │  **[M2] `apiGateway?: boolean` flag leaks API Gateway awareness into each middleware** ← blocker
│  │  
│  │  ```ts
│  │  genBadRequestErrorMiddleware({ apiGateway: true })
│  │  genInternalServiceErrorMiddleware({ apiGateway: true })
│  │  ```
│  │  
│  │  api gateway mode is a chain-level concern but is passed as a flag to each middleware individually. any new middleware that behaves differently in api gateway context must also accept this flag. this is scope leak — each middleware accumulates gateway knowledge it shouldn't own.
│  │  
│  │  a cleaner model: each middleware has two separate behaviors, exposed as two separate generators or a single generator that's parameterized cleanly at the chain level.
│  │  
│  │  ---
│  │  
│  │  **[M3] `genInternalServiceErrorMiddleware` repeats `getIsBadRequestError` from `genBadRequestErrorMiddleware`** ← nitpick
│  │  
│  │  ```ts
│  │  // in genInternalServiceErrorMiddleware.onError:
│  │  const isBadRequest = getIsBadRequestError({ error });
│  │  if (isBadRequest) return;
│  │  ```
│  │  
│  │  the internal service middleware only needs this guard because middy continues calling `onError` handlers even after one sets `request.response`. this is implicit coupling: if `genBadRequestErrorMiddleware` is removed from the chain, `genInternalServiceErrorMiddleware` still silently skips bad request errors.
│  │  
│  │  the guard belongs at the chain assembly level (or middy should stop the error chain once handled). as-is, the responsibility for "don't handle bad requests here" is duplicated.
│  │  
│  │  ---
│  │  
│  │  **[M4] `_genContext` parameter in `genLambdaEndpoint` is accepted but unused** ← nitpick
│  │  
│  │  ```ts
│  │  export const genLambdaEndpoint = <TInput, TOutput>(
│  │    config: { ... },
│  │    _genContext?: { log?: LogMethods },   // ← underscore-prefix = unused
│  │  ```
│  │  
│  │  the `_` prefix signals intentional non-use. the parameter exists in the public api surface, is exported via `GenLambdaEndpointContext`, but does nothing. either remove it (YAGNI) or use it (inject into ioLogger fallback when trail log unavailable).
│  │

│  │  
│  │  ### scope leaks
│  │  
│  │  **[S1] `ContextLogTrail` from sdk-logs leaks into public api via `EndpointOperation`** ← blocker
│  │  
│  │  ```ts
│  │  export type EndpointOperation<TInput, TOutput> = (
│  │    input: { event: TInput },
│  │    context: ContextLogTrail,   // ← from sdk-logs, not from this sdk
│  │  ) => Promise<TOutput>;
│  │  ```
│  │  
│  │  consumers must import from both `sdk-aws-lambda` and `sdk-logs` to type their handlers. `LambdaEndpointContext` exists in `domain.objects/` precisely to solve this — but it's not exported from `index.ts`, and `EndpointOperation` uses the raw sdk-logs type anyway.
│  │  
│  │  fix: export `LambdaEndpointContext` from index, use it in `EndpointOperation`.
│  │  
│  │  ---
│  │  
│  │  **[S2] `WrappedPayload` defined in two files with identical shape** ← nitpick
│  │  
│  │  - `getIsWrappedPayload.ts`: `interface WrappedPayload<TEvent>`
│  │  - `genLambdaEndpoint.ts`: `type WrappedPayload<TInput>`
│  │  
│  │  same structure, different file, both named the same. `genLambdaEndpoint.ts` should import from `getIsWrappedPayload.ts` (or a shared types file), not re-define.

│  │  // exported type:

│  │  these types are shared protocol shapes (used by both bad request and internal service middleware) but live in `genBadRequestErrorMiddleware.ts`. they should be in a shared types file or in `domain.objects/`, not inside a specific middleware's file.
