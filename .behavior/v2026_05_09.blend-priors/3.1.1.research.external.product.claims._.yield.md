## 1. middleware vs pure function handlers

### [OPIN] middleware pattern is best practice for lambda

> "Using middleware helps keep non-functional code outside your handler function, keeping the business logic in your handler pure and testable, while isolating and making all non-functional logic reusable." [1]

> "Middy is probably the most popular middleware framework in the AWS Lambda Node.js space." [1]

### [FACT] middleware follows onion-like execution

> "Before and error middlewares are pushed onto their stacks, meaning they execute in the order they are registered. After middlewares are unshifted onto their stacks, meaning they execute in **reverse order** of registration." [2]

### [SUMP] middleware overhead is acceptable

implicit assumption: the abstraction cost of middleware is worth the organizational benefit for cross-cutting concerns.

**source [1-2]:**
1. [practitioner] ServerlessFirst - Using middleware for cross-cutting concerns — https://serverlessfirst.com/lambda-middleware/
2. [official] Middy handling errors docs — https://middy.js.org/docs/intro/handling-errors

## 2. correlation id vs trace id

### [FACT] correlation id and trace id serve different purposes

> "A correlation ID is a single identifier that tracks one request end-to-end across all services — it stays the same from entry point to final response." [3]

> "A trace ID is the root identifier in distributed tracing (OpenTelemetry, Jaeger, Zipkin) that groups all spans across services into one trace tree." [3]

### [OPIN] use both together

> "You can use both together: the correlation ID for log correlation, the trace ID for deep performance analysis." [3]

### [KHUE] which pattern fits sdk-aws-lambda?

should sdk-aws-lambda use correlation IDs (simpler, broader retention) or trace IDs (more structured, better tooling)?

**source [3]:**
3. [practitioner] Last9 - Trace ID vs Correlation ID — https://last9.io/blog/correlation-id-vs-trace-id/

## 3. zod vs joi for validation

### [FACT] zod has native typescript type inference

> "With Zod, you can automatically create types for you using z.infer<>, so you don't need to write types separately." [4]

> "Joi has been around longer and has richer validation rules, but does not infer TypeScript types natively." [4]

### [FACT] zod is smaller than joi

> "Zod has a bundle size of 54KB while Joi is 145KB." [4]

### [OPIN] new typescript projects should default to zod

> "New TypeScript projects should default to Zod. It's the ecosystem standard, has the best TypeScript integration, and is what most library authors assume you're using." [4]

### [FACT] zod v4 has major performance improvements

> "Benchmarks show 14x faster string parsing, 7x faster array parsing, and 6.5x faster object parsing compared to Zod 3." [5]

**sources [4-5]:**
4. [practitioner] BetterStack - Joi vs Zod comparison — https://betterstack.com/community/guides/scaling-nodejs/joi-vs-zod/
5. [practitioner] InfoQ - Zod v4 announcement — https://www.infoq.com/news/2025/08/zod-v4-available/

## 4. sdk client initialization

### [FACT] initialize clients outside handler

> "Initialize SDK clients and database connections outside of the function handler, and cache static assets locally in the /tmp directory." [6]

> "Initialize SDK clients outside of the Lambda handler method so that if the execution context is reused, the initialization of the service client can be skipped." [7]

### [FACT] aws sdk v3 reduces cold start via modular imports

> "V2 imports the entire SDK (~70 MB). V3 lets you import only what you need. This is crucial for Lambda because smaller bundle sizes directly improve cold start performance." [8]

### [SUMP] container reuse is common enough to optimize for

implicit assumption: lambda execution environments are reused frequently enough that warm start optimization is worthwhile.

8. [official] AWS SDK v2 to v3 migration — https://docs.aws.amazon.com/sdk-for-javascript/v3/developer-guide/migrating.html

## 5. lambda-to-lambda invocation patterns

### [OPIN] direct lambda-to-lambda is anti-pattern

> "Avoid anti-patterns like Lambda functions calling other Lambda functions, and instead leverage services like Amazon SQS, SNS, or Step Functions." [9]

### [FACT] direct invocation has cost implications

> "Invoking a Lambda function from another one doubles the cost because each execution has its pricing." [9]

### [FACT] direct invocation creates tight coupling

> "In nested invocations, error handling can become much more complex. The concurrency of all three functions must be equal." [9]

### [KHUE] when is direct invocation acceptable?

sdk-aws-lambda provides askLambdaEndpoint for direct invocation. when is this pattern appropriate vs SQS/SNS?

> "At what point is an AWS SQS queue overkill as a Lambda function's source?" [9]

### [SUMP] some use cases require synchronous invocation

implicit assumption: despite anti-pattern guidance, synchronous lambda-to-lambda invocation is sometimes necessary (e.g., request-response patterns where the caller needs immediate result).

**source [9]:**
9. [official] AWS blog - Lambda anti-patterns in event-driven architectures — https://aws.amazon.com/blogs/compute/operating-lambda-anti-patterns-in-event-driven-architectures-part-3/

## 6. input validation timing

### [OPIN] validate early, fail fast

> "If the event data is invalid, a ValidationError will be raised, and the function will fail early with a descriptive error response." [10]

> "Early validation with Pydantic also facilitates runtime improvements, such as faster failure for invalid inputs, reduced load and execution costs, and improved security against malicious incoming data." [10]

### [FACT] validation prevents retry cost explosion

> "The consequences of not validating early can be significant: your entire batch (processed and unprocessed records alike) will be returned to the queue, ready to be processed all over again." [10]

### [KHUE] api gateway validation vs function validation?

> "API Gateway only covers one entry point. Validating inside the function means you're protected regardless of how the function gets called." [10]

**source [10]:**
10. [practitioner] Medium - AWS Lambda Event Validation — https://medium.com/cyberark-engineering/aws-lambda-event-validation-from-zero-to-hero-2ca950acd2ea

## 7. error response sanitization

### [OPIN] sanitize error responses for clients

> "Developers should catch all exceptions in the function_handler, log them and return a sanitized and gentle response to the requester." [11]

> "It is a good practice to avoid the code from raising exceptions, since it will expose internal implementation details which could potentially compromise the application security." [11]

### [FACT] lambda returns stack traces by default

> "When errors occur, Lambda returns an errorMessage, errorType, and stackTrace (a list of string expressions showing the stack trace)." [12]

### [SUMP] stack traces should not reach external clients

implicit assumption: exposing stack traces to external clients is a security risk. internal logs should capture full details.

**sources [11-12]:**
11. [practitioner] EdgeDelta - Lambda Log guide — https://edgedelta.com/company/knowledge-center/lambda-logging
12. [official] AWS API Gateway - Handle Lambda errors — https://docs.aws.amazon.com/apigateway/latest/developerguide/handle-errors-in-lambda-integration.html

## 8. type safety in serverless

### [FACT] typescript cannot validate runtime data

> "While TypeScript can guarantee type safety within your codebase, it cannot inherently protect you from invalid data entering your application at runtime." [13]

### [OPIN] runtime validation is essential

> "Runtime validation steps in to fill this critical gap. Tools like Zod bridge the gap between compile-time type checks and actual data validation at runtime." [13]

### [OPIN] combine compile-time and runtime validation

> "TypeScript provides a strong safety net, ensuring that data passed between services (or received from event sources) adheres to expected contracts." [13]

> "The debate essentially centers on TypeScript's compile-time guarantees being necessary but insufficient for production serverless applications—runtime validation remains essential." [13]

**source [13]:**
13. [tutorial] Steve Kinney - Type Safety vs Runtime Validation — https://stevekinney.com/courses/full-stack-typescript/type-safety-vs-runtime-validation

## 10. cors configuration

### [FACT] http api handles cors at gateway level

> "With HTTP APIs, the CORS headers are added by the gateway itself, so you don't need to add them in your Lambda function." [15]

### [FACT] rest api with proxy integration requires lambda cors

> "With Lambda Proxy Integration, API Gateway passes the entire request to Lambda and returns the Lambda response directly to the client. In this mode, Lambda must return CORS headers in its response." [15]

### [OPIN] avoid wildcard cors in production

> "Using Access-Control-Allow-Origin: * allows any website to access your resources. Always specify exact origins in production." [15]

## conflicts and convergence

### convergence: validate early

sources [10], [13] converge: validate input as early as possible, fail fast.

### convergence: zod for new typescript projects

sources [4], [5] converge: zod is the default choice for new typescript projects.

### convergence: init clients outside handler

sources [6], [7], [8] all agree: initialize SDK clients outside handler for reuse.

### conflict: direct lambda invocation

source [9] warns against lambda-to-lambda, but sdk-aws-lambda provides askLambdaEndpoint for exactly this purpose.

**resolution:** document when direct invocation is appropriate (synchronous request-response) vs when to use queues (async fire-and-forget).

### anti-patterns

> "Invoking a Lambda function from another one doubles the cost." — source [9] warns against lambda chains.

> "Using Access-Control-Allow-Origin: * allows any website to access your resources." — source [15] warns against wildcard CORS.

> "Stack traces can expose internal implementation details which could potentially compromise application security." — source [11] warns against error exposure.
