# reflection: premortem

> how could this fail?

---

## premortem: imagine we shipped and it failed

project forward: we shipped sdk-aws-lambda, and it failed badly. why?

### most plausible failure scenarios

1. **migration abandonment** — developers start to migrate from simple-lambda-handlers/simple-lambda-client but hit too many break changes. they give up and stay on old packages. adoption stalls.

2. **trail propagation regression** — exids don't propagate correctly across service boundaries. production debug becomes impossible. devops can't trace requests. incident response time increases from minutes to hours.

3. **type safety false confidence** — zod schemas compile but don't match runtime behavior. developers trust types, deploy, and hit runtime errors. trust in the package erodes.

4. **middleware chain breakage** — middy v7 behaves differently from v1 in subtle ways. error handler, execution order, or context injection breaks. handlers fail silently or with cryptic errors.

5. **aws sdk v3 edge cases** — InvokeCommand response handler differs from v2. payload parse, error detection, or timeout behavior breaks. askLambdaEndpoint returns wrong data or throws unexpected errors.

---

## what obvious signs did we ignore?

look back from imagined failure — what signs were present that we dismissed?

- **six major versions of middy** — "it's just middleware, how different can it be?" but the signature changed fundamentally (callback to request object)

- **aws sdk v3 is a rewrite, not an upgrade** — "it's the same AWS API" but client instantiation, promise handler, and error shapes all differ

- **joi to zod is not 1:1** — "both do validation" but error messages, coercion rules, and edge cases differ

- **two repos existed for a reason** — "just merge them" but the separation may have prevented couple issues we'll reintroduce

- **no integration tests with real lambdas** — "unit tests are enough" but cross-boundary behavior is where bugs hide

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| middy v7 middleware is compatible | middleware chain breaks, errors don't propagate, handlers fail | medium |
| aws sdk v3 response shapes match v2 | payload parse fails, error detection misses cases | medium |
| zod validation matches joi behavior | edge cases behave differently, validation passes when shouldn't | medium |
| trail propagation logic is straightforward | exids lost in async boundaries, logs become uncorrelated | low |
| developers will read migration guide | they'll npm install and expect it to work, file issues | high |
| cache dedup behavior is preserved | responses cached incorrectly, stale data returned | low |
| api gateway v1/v2 differences are handled | some deployments break based on gateway type | medium |

## what is the nightmare scenario?

**production services fail after upgrade, with no way to debug.**

scenario:
1. team upgrades svc-jobs to sdk-aws-lambda
2. deployment succeeds, tests pass
3. production traffic starts to flow
4. intermittent errors appear — askLambdaEndpoint returns null sometimes
5. no exids in logs — trail propagation broke silently
6. rollback fails — database schema changed in deployment
7. services degraded for hours
8. root cause: aws sdk v3 returns `Payload` as Uint8Array, not Buffer — parse logic fails silently

**why hardest to recover from:**
- silent failures are worst — no errors, just wrong data
- absent trail means can't trace what happened
- production traffic continues to flow, bad state accumulates

## what mitigations to consider?

| risk | mitigation | cost of mitigation |
|------|------------|--------------------|
| migration abandonment | provide before/after examples for every pattern | 2-3 days documentation |
| trail propagation bugs | integration tests with real lambdas that verify exid flow | 1-2 days test setup |
| type safety gaps | unit tests for every zod schema edge case | 1 day per schema |
| middy v7 incompatibilities | read middy upgrade guides, test middleware chain exhaustively | 1 day research + tests |
| aws sdk v3 edge cases | test payload parse with real lambda responses | 1 day integration tests |
| cache dedup regression | preserve extant cache tests, add new edge cases | 0.5 days |
| api gateway format issues | test both v1 and v2 event formats | 0.5 days |
| silent failures | add explicit error throws for unexpected states (fail loud) | built into design |
| developers skip docs | make happy path work without read docs, docs for edge cases | careful API design |

## summary

the highest risks are **silent failures in production** — especially trail propagation loss and aws sdk v3 payload parse. mitigations:

1. **integration tests with real lambdas** — not just unit tests with mocks
2. **fail-fast design** — explicit errors for unexpected states
3. **exhaustive middleware chain tests** — verify middy v7 behavior matches expectations
4. **clear migration guide** — reduce abandonment risk
5. **type tests for zod schemas** — verify compile-time types match runtime
