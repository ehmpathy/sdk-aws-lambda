# research: internal prod codepaths

> patterns from simple-lambda-handlers and simple-lambda-client

---

## summary

this research inventories all production patterns from the two source repos:
- **simple-lambda-handlers** — handler generation with middleware
- **simple-lambda-client** — lambda invocation with cache

each pattern is marked:
- **[REUSE]** — keep as-is in sdk-aws-lambda
- **[EXTEND]** — keep but enhance
- **[REPLACE]** — remove and replace with new approach

---

## 1. simple-lambda-handlers patterns

### 1.1 createStandardHandler

**file:** `src/contract/createStandardHandler.ts` [1]

> "return middy(logic).use(badRequestErrorMiddleware()).use(internalServiceErrorMiddleware({ logError: logMethods.error })).use(ioLoggingMiddleware({ logDebug: logMethods.debug, logTranslate })).use(joiEventValidationMiddleware({ schema }))" [1:46-62]

**[EXTEND]** — core pattern of middleware composition is sound. rename to `genLambdaEndpoint` per wish. replace Joi validation with Zod. add trail injection via sdk-logs.

**relation to wish:**
- maps directly to `genLambdaEndpoint({ schema, invoke })`
- `logic` param becomes `invoke`
- `schema` param now expects Zod schema, not Joi
- `log` param becomes optional context, auto-generated with trail

**relation to wish:**

### 1.5 badRequestErrorMiddleware

**file:** `src/logic/middlewares/badRequestErrorMiddleware.ts` [4]

**relation to wish:**

- enables proper error response sanitization for API Gateway (no stack traces)

### 1.7 ioLogMiddleware

## 2. simple-lambda-client patterns

### 2.1 invokeLambdaFunction

**file:** `src/invokeLambdaFunction.ts` [11]

> "export const invokeLambdaFunction = async <O = any, I = any>({ service: serviceName, function: functionName, stage, event, logDebug, cache }: { service: string; function: string; stage: string; event: I; logDebug?: LogMethod; cache?: SimpleCache<O>; }): Promise<O>" [11:24-38]

**[EXTEND]** — rename to `askLambdaEndpoint`. change param names per wish (`service` + `function` → `which`, `event` → `with`). make stage derived from context.env.STAGE. inject trail from context.log.

**relation to wish:**
- maps to `askLambdaEndpoint({ which: { service, function }, with }, context)`
- stage derived from context.env.STAGE
- trail.exid extracted from context.log

---

### 2.2 executeLambdaInvocation

**file:** `src/executeLambdaInvocation.ts` [12]

> "const lambda = new Lambda(); ... const response = await lambda.invoke({ FunctionName: lambdaName, Payload: JSON.stringify(event) }).promise()" [12:1-42]

**[REPLACE]** — uses aws-sdk v2 (deprecated). replace with @aws-sdk/client-lambda v3 command pattern: `client.send(new InvokeCommand({ ... }))`.

**relation to wish:**
- core invocation logic, must migrate to v3
- keep function name construction: `[service, stage, function].join('-')`

---

**file:** `src/errors.ts` [13]

**[EXTEND]** — good error class with lambda name, response details, and event. add trail.exid for correlation.

**relation to wish:**
- satisfies error classification (LambdaInvocationError vs ValidationError)
- includes target service/function for debug

---

### 2.4 UnsuccessfulStatusCodeError

**file:** `src/errors.ts` [13]

> "export class UnsuccessfulStatusCodeError extends Error { constructor({ code, payload }: { code: number | undefined; payload: any }) { const message = `Status code does not indicate success: ${code}; ${JSON.stringify(payload)}`; super(message); } }" [13:3-9]

**[REUSE]** — simple error for non-200 status codes. keep as-is.

**relation to wish:**
- thrown when lambda.invoke returns non-200 status
- distinct from LambdaInvocationError (which is for error payloads)

---

### 2.5 getSimpleLambdaClientCacheKey

**file:** `src/cache/getSimpleLambdaClientCacheKey.ts` [14]

> "const idealKey = [[serviceName, stage, functionName].join('-'), JSON.stringify(event)].join('.'); ... const uniquePart = asHashSha256Sync(JSON.stringify(idealKey)); return [humanPart, uniquePart].join('.')" [14:17-32]

**[REUSE]** — cache key includes service, stage, function, and event hash. human-readable prefix + SHA256 for uniqueness. keep this approach.

**relation to wish:**
- enables dedup via with-simple-cache
- cache key construction is sound

---

### 2.6 globalSyncCache for dedup

**file:** `src/invokeLambdaFunction.ts` [11:17-19]

> "const globalSyncCache = createCache({ expiration: { seconds: 15 } })" [11:17-19]

**[REUSE]** — synchronous cache for dedup of parallel requests. 15s expiration matches expected lambda timeout. keep pattern.

**relation to wish:**
- prevents duplicate invocations for same request
- integrates with with-simple-cache

---

### 2.7 withSimpleCacheAsync integration

**file:** `src/invokeLambdaFunction.ts` [11:40-53]

> "const execute = cache ? withSimpleCacheAsync(executeLambdaInvocation, { cache: { output: cache, deduplication: globalSyncCache }, serialize: { key: (input) => getSimpleLambdaClientCacheKey(...) } }) : executeLambdaInvocation" [11:40-53]

**[REUSE]** — conditional cache based on whether cache was provided. uses dedup cache + output cache. keep pattern.

**relation to wish:**
- cache is optional parameter
- same cache behavior as simple-lambda-client
