## 1. middy v1.5.2 → v7 migration

### [FACT] middy v7 requires node.js 18+

> "middy v7 requires Node.js 18 or greater." [1]

this is a hard requirement — lambda runtime must be nodejs18.x or higher.

### [FACT] v7 is ESM-first with CJS compatibility

> "middy v7 is an ESM-first module, but also supports CommonJS (CJS) for backwards compatibility." [1]

### [FACT] middleware signature changed significantly

> "middy v1 used `(handler, next) => ...` pattern. middy v7 uses `(request) => { ... return handler(request) }` pattern with async/await throughout." [2]

### [FACT] before/after hooks replaced by request object

> "The request object contains `event`, `context`, `response`, `error`, and `internal` properties. Middlewares modify this object directly." [1]

### [FACT] error handling changed

> "In v7, error middlewares receive the request object with the error property set. The `onError` hook is now part of the middleware chain, not a separate callback." [2]

### [OPIN] migration requires middleware rewrite

> "If you have custom middleware written for middy v1-v4, you'll need to rewrite them for v5+ due to the new signature." [3]

### [KHUE] what about durable-execution-sdk peerdep?

simple-lambda-handlers has `@dazn/lambda-powertools-middleware-correlation-ids` which may have peer dependency conflicts with middy v7.

**sources [1-3]:**
1. [official] Middy v7 documentation — https://middy.js.org/docs/intro/getting-started
2. [official] Middy upgrade guide v4→v5 — https://middy.js.org/docs/upgrade/4-5
3. [practitioner] Serverless middleware patterns — https://serverlessfirst.com/lambda-middleware/

## 2. aws sdk v2 → v3 migration

### [FACT] v3 uses modular architecture

> "The AWS SDK for JavaScript v3 is a rewrite of v2 with some great new features, including modular architecture." [4]

### [FACT] v3 reduces bundle size significantly

> "V2 imports the entire SDK (~70 MB). V3 lets you import only what you need. This is crucial for Lambda because smaller bundle sizes directly improve cold start performance." [5]

### [FACT] v3 has different client instantiation

> "In v2, you created clients with `new AWS.Lambda()`. In v3, you import and instantiate specific clients: `import { LambdaClient } from '@aws-sdk/client-lambda'`." [4]

### [FACT] v3 uses command pattern

> "V3 uses a command pattern where you create command objects and send them through the client: `client.send(new InvokeCommand(params))`." [4]

### [FACT] aws provides codemod for migration

> "AWS provides a codemod tool to automate much of the migration: `npx aws-sdk-js-codemod -t v2-to-v3`." [6]

### [SUMP] v2 is deprecated but still supported

> "AWS SDK for JavaScript v2 entered maintenance mode on September 8, 2024. AWS will continue to provide critical bug fixes and security patches for v2 through September 8, 2025." [7]

implicit assumption: migration to v3 is not urgent but recommended.

### [FACT] promise handling differs

> "V2 used `.promise()` method. V3 returns promises natively — no `.promise()` call needed." [4]

### [KHUE] middleware in v3?

> "V3 introduces a middleware stack for request/response handling, different from v2's event-based approach." [8]

should we leverage v3 middleware for trail injection, or handle separately?

## convergence

### both migrations require node.js 18+

middy v7 and modern lambda best practices converge on node.js 18+ as minimum runtime.

### both migrations improve bundle size

- middy v7: smaller, more focused middleware
- aws sdk v3: modular imports reduce bundle significantly

---

## conflicts

### middleware approach differs

middy v7 uses request object pattern; aws sdk v3 uses command/middleware pattern. these are complementary but require understanding both.

---

## migration strategy for sdk-aws-lambda

### clean break recommended

given both libraries have major breaking changes (6 major versions for middy, new architecture for aws sdk), a clean implementation is preferable to incremental migration.

### drop compatibility shims

- no middy v1 compatibility layer
- no aws-sdk v2 compatibility layer
- require node.js 18+ runtime

### key changes

| component | before | after |
|-----------|--------|-------|
| middy | v1.5.2 | v7.5.0 |
| aws-sdk | v2.1426.0 | @aws-sdk/client-lambda v3 |
| node.js | 14+ | 18+ |
| middleware | callback-based | async/await |
| lambda invoke | `lambda.invoke().promise()` | `client.send(new InvokeCommand())` |
