# vision: sdk-aws-lambda

> merge simple-lambda-handlers + simple-lambda-client into one unified sdk

---

## the outcome world

### before

```
repo: simple-lambda-handlers
├── createApiGatewayHandler()
├── createStandardHandler()
└── Joi validation, Middy middleware

repo: simple-lambda-client
├── invokeLambdaFunction()
└── caching, error handling
```

two repos. two release cycles. handler changes require client changes, but they ship separately. trail context? manually threaded. schema validation? Joi on handler, none on client.

**the pain:**
- add a field to an endpoint → update handler, release, then update client, release
- trail.exid threading is manual and error-prone
- client has no type safety against the handler's actual schema
- Joi is dated; zod is the standard now
- middy v1.5.2 is ancient (v7.5.0 is current)

### after

```
repo: sdk-aws-lambda
├── genLambdaEndpoint        # define
│   ├── .forApiGateway()
│   └── .forSqs()
├── askLambdaEndpoint()      # invoke
└── shared: zod schemas, trail threading, types
```

one repo. one release. handler and client evolve together.

**the aha moment:**

```ts
// handler: svc-jobs/getJobByUuid.ts
export const handler = genLambdaEndpoint({
  schema: {
    input: z.object({ uuid: z.string().uuid() }),
    output: z.object({ job: JobSchema }),
  },
  invoke: async (event, context) => {
    context.log.debug('lookup job', { uuid: event.uuid }); // trail.exid already in log
    const job = await daoJob.findByUuid({ uuid: event.uuid });
    return { job };
  },
});

// client: anywhere
const { job } = await askLambdaEndpoint({
  which: { service: 'svc-jobs', function: 'getJobByUuid' },
  with: { uuid: '123' },
}, context); // trail auto-extracted from context.log, injected into call
```

- change the schema → types update for both handler AND client
- trail.exid flows automatically across service boundaries
- zod validates on both ends
- one PR, one release

## user experience

### usecase 1: define a lambda endpoint

**goal:** create a new lambda function with validation, logging, error handling

**before (simple-lambda-handlers):**
```ts
import { createStandardHandler } from 'simple-lambda-handlers';
import Joi from 'joi';

const schema = Joi.object({ uuid: Joi.string().uuid().required() });

export const handler = createStandardHandler({
  logic: async (event, context) => {
    // no log with trail context
    // must manually pass log around
    return { job };
  },
  schema,
  log: console, // no trail integration
});
```

**after (sdk-aws-lambda):**
```ts
import { genLambdaEndpoint } from 'sdk-aws-lambda';
import { z } from 'zod';

export const handler = genLambdaEndpoint({
  schema: {
    input: z.object({ uuid: z.string().uuid() }),
    output: z.object({ job: JobSchema }),
  },
  invoke: async (event, context) => {
    // context.log already has trail.exid from caller
    context.log.debug('lookup job', { uuid: event.uuid });
    return { job };
  },
});
```

**timeline:**
1. define zod schemas for input/output
2. implement invoke function with (input, context) pattern
3. export handler — done

### usecase 2: invoke a lambda from another service

**goal:** call svc-jobs from svc-invoices with trail threading

**before (simple-lambda-client):**
```ts
import { invokeLambdaFunction } from 'simple-lambda-client';

const job = await invokeLambdaFunction({
  service: 'svc-jobs',
  function: 'getJobByUuid',
  stage: process.env.STAGE,
  event: { uuid },
  logDebug: log.debug, // manual
});
// no type safety on event or response
// trail.exid? hope you remembered to include it in the event
```

**after (sdk-aws-lambda):**
```ts
import { askLambdaEndpoint } from 'sdk-aws-lambda';

const { job } = await askLambdaEndpoint<GetJobResponse, GetJobRequest>({
  which: { service: 'svc-jobs', function: 'getJobByUuid' },
  with: { uuid },
}, context);
// typed request/response
// trail auto-extracted from context.log, injected into call
// stage derived from context.env
```

**timeline:**
1. import askLambdaEndpoint
2. specify which endpoint + payload
3. pass log context — trail threading is automatic

### usecase 3: api gateway endpoint with CORS

**goal:** create an http endpoint with CORS, body parsing, security headers

**before:**
```ts
import { createApiGatewayHandler, HttpStatusCode } from 'simple-lambda-handlers';

export const handler = createApiGatewayHandler({
  logic: async ({ body, headers }) => ({
    statusCode: HttpStatusCode.SUCCESS_200,
    body: result,
  }),
  schema,
  log,
  cors: { origins: ['https://app.example.com'], withCredentials: true },
});
```

**after:**
```ts
import { genLambdaEndpoint } from 'sdk-aws-lambda';

export const handler = genLambdaEndpoint.forApiGateway({
  schema: { input: BodySchema, output: ResponseSchema },
  invoke: async (event, context) => result,
  cors: { origins: ['https://app.example.com'], credentials: true },
});
```

### usecase 4: introspection mode

**goal:** discover endpoint schemas without invoking

**new capability:**
```ts
// special event triggers introspection
const schemas = await askLambdaEndpoint({
  which: { service: 'svc-jobs', function: 'getJobByUuid' },
  with: { __introspect: true },
}, context);

// returns:
// {
//   input: { type: 'object', properties: { uuid: { type: 'string', format: 'uuid' } } },
//   output: { type: 'object', properties: { job: { ... } } },
// }
```

enables: auto-generated api docs, client codegen, schema registry

## mental model

### how users describe it

> "it's like having one sdk for both sides of the lambda call"

> "the endpoint defines the contract, and the client knows about it"

> "trail context just works — i don't think about it anymore"

### analogies

| analogy | maps to |
|---------|---------|
| tRPC | genLambdaEndpoint = router, askLambdaEndpoint = client |
| gRPC | schema = .proto, introspection = reflection |
| REST client | askLambdaEndpoint = axios with types |

### vocabulary

| user term | sdk term | meaning |
|-----------|----------|---------|
| "endpoint" | genLambdaEndpoint | the handler definition |
| "call" / "invoke" | askLambdaEndpoint | invoking from client |
| "schema" | zod schemas | input/output validation |
| "trail" | trail.exid | request correlation id |
| "trigger" | .forApiGateway, .forSqs | event source adapter |

## evaluation

### how well does it solve the goals?

| goal | score | notes |
|------|-------|-------|
| single repo for handler+client | ✅ | one repo, coordinated releases |
| trail threading | ✅ | auto-extracted from log, injected into call |
| schema validation | ✅ | zod on both ends |
| typed client | ✅ | generics on askLambdaEndpoint |
| modernization | ✅ | zod > joi, (input, context) pattern |
| introspection | ✅ | new capability |

### pros

- **coordinated releases** — handler + client changes in one PR
- **trail auto-threading** — no manual exid passing
- **type safety** — schemas define the contract, types flow to client
- **modern stack** — zod, (input, context), sdk-logs integration
- **introspection** — runtime schema discovery
- **pit of success** — harder to forget trail, harder to mismatch types

### cons

- **migration effort** — extant services use old packages
- **breaking changes** — api surface changes significantly
- **zod dependency** — joi users must migrate
- **bundle size** — single package vs two smaller ones
- **learning curve** — new patterns for extant users

### edgecases and pit of success

| edgecase | pit of success solution |
|----------|------------------------|
| forgot to pass log context | default log auto-generated with trail |
| schema mismatch client/handler | compile-time error if types exported |
| wrong stage | derived from env.STAGE by default |
| missing trail.exid | auto-generated if not present |
| invalid input | zod validates before invoke runs |
| unhandled error | auto-wrapped, logged, classified |

## groundwork

### external research

**aws lambda invoke api:**
- verified: Lambda.invoke() returns `{ StatusCode, Payload, FunctionError }`
- error detection: check for `errorMessage`, `errorType`, `stackTrace` in payload
- constraint: Payload is Buffer, must parse JSON

**zod-to-json-schema:** ✅ verified
- exists: `zod-to-json-schema` v3.25.2 (npm)
- description: "Converts Zod schemas to Json Schemas"
- peerDependencies: `zod: '^3.25.28 || ^4'` — zod v4 supported
- enables introspection mode returning json schema from zod definitions

### internal research

**simple-lambda-handlers exports** (researched via git.repo.get):
- `createApiGatewayHandler` — src/contract/createApiGatewayHandler.ts:114
- `createStandardHandler` — src/contract/createStandardHandler.ts:10
- `HttpStatusCode` — src/domain/constants.ts:9
- `HandlerLogic`, `EventSchema`, `LogMethods` — src/domain/general.ts
- middleware chain: badRequestError → internalServiceError → ioLogging → validation
- uses Joi for validation, Middy v1 for middleware

**simple-lambda-client exports** (researched via git.repo.get):
- `invokeLambdaFunction` — src/invokeLambdaFunction.ts:24
- `LambdaInvocationError` — src/errors.ts:11
- `UnsuccessfulStatusCodeError` — src/errors.ts:3
- `getSimpleLambdaClientCacheKey` — src/cache/getSimpleLambdaClientCacheKey.ts:6
- uses aws-sdk v2, with-simple-caching for dedup

**sdk-logs trail pattern:** ✅ verified via git.repo.get
- `LogTrail` interface: `{ exid: string | null; stack: string[] }`
- `exid` = external identifier for request correlation across services
- `stack` = procedure call stack, appended by `withLogTrail()` wrapper
- `genContextLogTrail({ trail, env })` creates log methods with trail auto-injected
- `context.log.trail` extracts current trail state
- `withLogTrail(logic, { name })` preserves exid, appends name to stack
- source: ehmpathy/sdk-logs src/domain.objects/LogTrail.ts, src/domain.operations/genContextLogTrail.ts

## what is awkward?

### naming: gen vs create vs define

`genLambdaEndpoint` follows gen/set/get convention, but "gen" typically means "find or create". this is always "create". consider:
- `defLambdaEndpoint` — define
- `newLambdaEndpoint` — new instance
- `lambdaEndpoint` — noun, then call it

### naming: ask vs invoke vs call

`askLambdaEndpoint` — intentional choice:
- "ask" implies optional response — correct, caller may not need response
- "invoke" is aws terminology but implies mandatory response
- "call" is generic

verdict: keep `askLambdaEndpoint`

### the .for.* pattern

```ts
genLambdaEndpoint.forApiGateway({ schema, invoke, cors })
```

is this a builder? does it return a new handler? does it mutate?

clearer alternatives:
```ts
genApiGatewayEndpoint({ schema, invoke, cors })
genLambdaEndpoint({ schema, invoke, trigger: 'apiGateway', cors })
```

### introspection via magic event

```ts
{ __introspect: true }
```

is awkward because:
- pollutes the event schema
- what if someone's schema has `__introspect`?
- how does zod validate it?

alternatives:
- separate function: `getLambdaEndpointSchema({ which })`
- header-based: `{ headers: { 'x-introspect': true } }`
- different invoke type: `InvocationType: 'Introspect'`

### cache location

verdict: keep on `askLambdaEndpoint`, same as simple-lambda-client (via with-simple-cache)

### backwards compatibility

**client-side (askLambdaEndpoint):** hard break, no shim — clean migration

**handler-side (genLambdaEndpoint):** backwards compatible with legacy callers

handlers MUST accept payloads from callers who haven't upgraded:

| caller type | payload format | trail source |
|-------------|----------------|--------------|
| sdk-aws-lambda client | `{ ...event, trail: { exid } }` | extract from `event.trail.exid` |
| legacy client (simple-lambda-client) | `event` (raw, no trail wrapper) | generate new exid |
| external trigger (SQS, SNS, etc.) | `event` (no trail control) | generate new exid |
| manual invoke (aws console, cli) | `event` (no trail) | generate new exid |

**invariant:** genLambdaEndpoint must never fail due to absent trail. if trail is absent, generate a fresh exid and continue. this ensures:
- incremental upgrades work (old clients → new handlers)
- external triggers work (SQS events have no trail)
- manual test invocations work (aws console, cli)
