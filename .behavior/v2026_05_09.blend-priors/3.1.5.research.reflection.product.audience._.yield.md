# reflection: audience

> who is this for and why do they care?

---

## who is the audience?

### primary: ehmpathy service developers

typescript developers who build lambda-based microservices within the ehmpathy ecosystem.

**context:**
- work daily with lambda handlers and cross-service invocations
- familiar with (input, context) pattern from other ehmpathy packages
- expect trail/exid propagation for observability
- maintain services like svc-jobs, svc-invoices, svc-notifications

**constraints:**
- must coordinate changes across handler and client code
- currently split across two repos (simple-lambda-handlers, simple-lambda-client)
- need to keep services running during migration

### secondary: devops and platform engineers

engineers who deploy and monitor lambda-based services.

**context:**
- configure lambda functions, API Gateway, SQS triggers
- monitor CloudWatch logs and traces
- debug production issues using exids

**constraints:**
- need clear error messages with correlation ids
- expect consistent log formats across services
- require observability without code changes

### tertiary: new team members

developers onboarding to ehmpathy codebases.

**context:**
- learning ehmpathy patterns and conventions
- need clear documentation and examples
- may come from different lambda/serverless backgrounds

**constraints:**
- unfamiliar with trail/exid concepts initially
- may expect patterns from other frameworks (Serverless Framework, SAM)
- need gentle learning curve

## why do they care?

### primary: ehmpathy service developers

| pain | how sdk-aws-lambda relieves it |
|------|-------------------------------|
| two repos to coordinate | single unified package |
| dated patterns (Joi, aws-sdk v2) | modern stack (Zod v4, @aws-sdk v3) |
| manual trail propagation | automatic trail threading |
| inconsistent handler patterns | standardized genLambdaEndpoint |
| type mismatches handler↔client | shared schema types |

**goal enabled:** write lambda services faster with confidence.

### secondary: devops and platform engineers

| pain | how sdk-aws-lambda relieves it |
|------|-------------------------------|
| missing correlation ids in logs | trail.exid auto-injected |
| unclear error sources | error classification with context |
| debug across service boundaries | trail propagation end-to-end |

**goal enabled:** debug production issues in minutes, not hours.

### tertiary: new team members

| pain | how sdk-aws-lambda relieves it |
|------|-------------------------------|
| split repos confuse onboarding | single package to learn |
| outdated docs in old repos | fresh documentation |
| inconsistent patterns | clear conventions |

**goal enabled:** productive contribution within first week.

## what do they care about?

### primary: ehmpathy service developers

**success criteria:**
- migration path from old packages is clear
- new API is intuitive and matches ehmpathy conventions
- type safety catches errors at compile time
- tests are easy to write and maintain

**acceptable tradeoffs:**
- breaking changes OK if migration is straightforward
- new dependencies OK if they're maintained and typed
- slight learning curve OK if payoff is significant

**delight vs disappointment:**
- delighted: handler + client in sync, less boilerplate, better errors
- disappointed: confusing API, poor docs, migration headaches

### secondary: devops engineers

**success criteria:**
- logs contain correlation ids by default
- errors include context for debug
- no observable behavior changes in deployed services

**acceptable tradeoffs:**
- code changes OK if logs improve
- new log format OK if CloudWatch queries work

### tertiary: new team members

**success criteria:**
- single package to npm install
- README explains the core concepts
- examples cover common cases

## how will they experience it?

### primary: ehmpathy service developers

**touchpoints:**
1. npm install sdk-aws-lambda (single package)
2. read migration guide (from simple-lambda-handlers, simple-lambda-client)
3. update imports and patterns in service code
4. run tests to verify behavior
5. deploy and monitor

**friction points:**
- breaking changes require code updates
- new API may need mental model shift
- tests may need updates for new mocks

**smooth experience:**
- clear migration guide with before/after examples
- codemods for common patterns if feasible
- tests pass after mechanical updates

### secondary: devops engineers

**touchpoints:**
1. observe new log formats in CloudWatch
2. use exid to trace requests
3. debug errors using context in logs

**friction points:**
- log format changes may break extant queries
- new error types may need dashboard updates

**smooth experience:**
- log format is backwards compatible or documented
- error types are predictable and documented

### tertiary: new team members

**touchpoints:**
1. read README and docs
2. look at example handlers
3. write first handler
4. write first cross-service call

**smooth experience:**
- README has quickstart in <5 minutes
- examples cover 80% of use cases
- types guide correct usage

---

## summary

sdk-aws-lambda primarily serves **ehmpathy service developers** who want a unified, modern package for lambda handlers and clients. success means:

1. single package replaces two
2. modern stack (Zod, aws-sdk v3, middy v7)
3. automatic trail propagation
4. clear migration from old packages
5. type-safe handler↔client contracts
