# research: external product access

> remote repositories, contracts, and best practices for sdk-aws-lambda

---

## summary

sdk-aws-lambda requires access to these remote boundaries:

| boundary | interface | purpose |
|----------|-----------|---------|
| AWS Lambda service | @aws-sdk/client-lambda | invoke lambda functions |
| API Gateway events | event conversion | parse http requests |
| Zod schemas | runtime validation | input/output validation |
| Middy middleware | @middy/core | middleware chain |
| CloudWatch logs | structured logging | observability |

## 1. aws lambda invocation via @aws-sdk/client-lambda

### contract

the aws sdk v3 lambda client provides `InvokeCommand` for lambda-to-lambda calls.

> "When using Lambda, you should use a single SDK client per service, per region, and initialize it outside of the handler's codepath to optimize for Lambda's container reuse." [1]

> "API calls themselves should be made from within the handler's codepath to ensure that API calls are signed at the very last step of Lambda's execution cycle." [1]

### response shape

> "Lambda.invoke() returns `{ StatusCode, Payload, FunctionError }`" — error detection: check for `errorMessage`, `errorType`, `stackTrace` in payload. Payload is Buffer, must parse JSON." [1]

> "The presence of an error in the response is indicated by a header named X-Amz-Function-Error." [2]

### best practices

> "Initialize SDK clients and database connections outside of the function handler, and cache static assets locally in the /tmp directory. Subsequent invocations processed by the same instance of your function can reuse these resources." [3]

> "V2 imports the entire SDK (~70 MB). V3 lets you import only what you need. This is crucial for Lambda because smaller bundle sizes directly improve cold start performance." [4]

### migration from v2 to v3

> "The aws-sdk-js-codemod collection of scripts helps migrate extant AWS SDK for JavaScript (v2) applications to use v3 APIs. You can run this with: `npx aws-sdk-js-codemod -t v2-to-v3 path/to/your/files`" [4]

> "In v3, invoking Lambda changes from `lambda.invoke({...}).promise()` to `lambda.send(new InvokeCommand({...}))`" [4]

## 2. middy v7 middleware framework

### contract

> "Powertools officially supports `@middy/core` both v4.x - v7.x, indicating that v7 is a current version being actively maintained." [5]

> "Middy can be used with TypeScript with typings built in in every official package. When using TypeScript, you should use what we call the Middleware-first, Handler-last approach." [6]

### middleware chain pattern

> "Middy comes with a very small core and an unobtrusive API to add the minimum amount of overhead to your code." [6]

> "You can define typed handlers using `middy<APIGatewayProxyEvent, APIGatewayProxyResult>()` and then chain your middlewares before calling the `.handler()` method." [6]

### cors middleware

> "The middleware sets HTTP CORS headers (Access-Control-Allow-Origin, Access-Control-Allow-Headers, Access-Control-Allow-Credentials), necessary for cross-origin requests, to the response object." [7]

> "Configuration options include: `credentials` (bool), `origins` (array), `headers` (string), `methods` (string), `exposeHeaders` (string), `maxAge` (string)." [7]

### input-output-logger middleware

> "By default, the logging operates by using the console.log function. You can pass as a parameter a custom logger with additional logic." [8]

> "Configuration options include logger function, executionContext boolean, lambdaContext boolean, and omitPaths string array to remove particular fields and redact sensitive data from logs." [8]

### validator middleware

> "It uses a compiled AJV validator to validate the input (request.event), context (request.context), and output (request.response) of the Lambda handler." [9]

## 3. correlation id and trail patterns

### pattern

> "Generate a correlation ID as early as possible in the request lifecycle, preferably at the entry point of your serverless application, such as API Gateway or application load balancer." [10]

> "Pass the correlation ID along with each request either as part of the custom header, body or metadata. Inject it into structured logs and pass it to downstream services or other Lambdas." [10]

### implementation

> "You can either manually capture and include correlation ID as part of your Lambda function logs or use tools like AWS Lambda Powertools. With Lambda Powertools, you can easily grab the correlation ID from predefined request path mapping." [10]

> "Lambda automatically generates a _X_AMZN_TRACE_ID value in the environment variable." [10]

### business impact

> "These strategies typically reduce Lambda logging expenses by 50–80%, cut troubleshooting time to 15–30 minutes, and speed up queries 10–100 times." [10]

**source [10]:**
10. [practitioner] EdgeDelta Lambda logging best practices, 2025 — https://edgedelta.com/company/knowledge-center/aws-lambda-logging-best-practices

## 4. zod schema validation

### contract

> "Zod is a TypeScript-first validation library that allows you to define schemas to validate data, from a simple string to a complex nested object." [11]

> "Zod lets you define a schema using native TypeScript types and validate the input while converting it to a first-class TypeScript object. You can use Zod to do both validation and define the type in one place." [12]

### zod v4 improvements

> "Zod version 4 delivers performance improvements, reduced bundle size, and updated API including a new simplified, tree-shakable mini package." [13]

> "Benchmarks show 14x faster string parsing, 7x faster array parsing, and 6.5x faster object parsing compared to Zod 3." [13]

> "The introduction of @zod/mini is a lightweight distribution weighing just ~1.9 KB gzipped, designed for tree-shakable validation in modern frontend applications." [13]

### lambda integration

> "Parse the request body with `safeParse()` method and return a 400 status code with validation errors if parsing fails." [12]

> "Middy-zod-validator middleware automatically validates incoming events using a given Zod schema and is especially useful for serverless projects using TypeScript." [14]

> "AWS Lambda Powertools parser is available as Middy.js middleware and TypeScript class method decorator, allowing you to specify a schema to parse the event." [15]

## 5. api gateway event formats

### v1 vs v2 format differences

> "Format 1.0 (REST API / v1) includes fields like `version: '1.0'`, `resource`, `path`, `httpMethod`, `headers`, `multiValueHeaders`, `queryStringParameters`, and `multiValueQueryStringParameters`." [16]

> "Format 2.0 (HTTP API / v2) includes fields like `version: '2.0'`, `routeKey`, `rawPath`, `rawQueryString`, `cookies`, and a simplified `headers` structure." [16]

> "Format 2.0 doesn't have multiValueHeaders or multiValueQueryStringParameters fields; duplicate headers are combined with commas and included in the headers field." [16]

### response format

> "In Lambda proxy integration, API Gateway requires the backend Lambda function to return output according to the following JSON format: `{ isBase64Encoded, statusCode, headers, multiValueHeaders, body }`" [17]

> "With the 2.0 format version, API Gateway can infer the response format, and functions can return a response with `statusCode`, `headers`, `body`, `isBase64Encoded`, and a new `cookies` field." [16]

**sources [16-17]:**
16. [official] AWS API Gateway Lambda proxy integration — https://docs.aws.amazon.com/apigateway/latest/developerguide/http-api-develop-integrations-lambda.html
17. [official] AWS API Gateway Lambda proxy integrations — https://docs.aws.amazon.com/apigateway/latest/developerguide/set-up-lambda-proxy-integrations.html

## 6. cors configuration

### best practices

> "Choose HTTP API unless you need REST API-specific features. With HTTP APIs, the CORS headers are added by the gateway itself, so you don't need to add them in your Lambda function." [18]

> "With Lambda Proxy Integration, API Gateway passes the entire request to Lambda and returns the Lambda response directly to the client. In this mode, Lambda must return CORS headers in its response." [18]

> "Using Access-Control-Allow-Origin: * allows any website to access your resources. Always specify exact origins in production." [18]

### error case handling

> "When API Gateway fails to invoke your Lambda function, it returns a server-side error, but Lambda cannot add the Access-Control-Allow-Origin header. Therefore, you need to configure API Gateway to add the CORS header for server-side errors." [18]

**source [18]:**
18. [official] AWS API Gateway CORS for REST APIs — https://docs.aws.amazon.com/apigateway/latest/developerguide/how-to-cors.html

## 7. error handling patterns

### structured errors

> "Create domain-specific error classes that include context about what went wrong and whether the error is retryable, such as ValidationError and DatabaseError classes with specific properties." [19]

> "You should return the same error signature for all types of errors so any frontend calling your APIs doesn't have to be concerned about the type of error they received." [19]

### response sanitization

> "If the function output is of a different format, API Gateway returns a 502 Bad Gateway error response." [17]

> "For Lambda functions that call out to third-party endpoints or other downstream resources, ensure that they can handle service errors and timeouts." [19]

**source [19]:**
19. [practitioner] bobbyhadz - AWS Lambda error handling — https://bobbyhadz.com/blog/aws-lambda-error-handling

## 8. idempotency and dedup

### pattern

> "A Lambda function is idempotent when it can be invoked multiple times with the same event with no risk of side effects. This is essential for distributed systems that rely on messaging services." [20]

> "In-memory cache is local to each Lambda execution environment and can be enabled with the use_local_cache parameter in IdempotencyConfig." [20]

### cache key design

> "By default, the idempotency key is a combination of Lambda function name, fully qualified name of your function, and a hash of the entire payload or part(s) of the payload you specify." [20]

### powertools integration

> "Powertools sees the COMPLETED record and short-circuits with the cached response — so no second publish happens." [20]

> "You can use the idempotency utility from Powertools for AWS Lambda to make lambda functions idempotent, which is available for Python, JavaScript (TypeScript), Java, and .Net." [20]

**source [20]:**
20. [official] AWS blog - Lambda idempotency with Powertools — https://aws.amazon.com/blogs/compute/handling-lambda-functions-idempotency-with-aws-lambda-powertools/

## conflicts and convergence

### convergence: client init outside handler

sources [1], [3], [4] all agree: initialize SDK clients outside the handler function.

### convergence: correlation id propagation

sources [10], [21] converge on using structured logging with correlation IDs for observability.

### convergence: zod for validation

sources [12], [14], [15] all recommend Zod for Lambda input validation, with Powertools providing official integration.

### conflict: cors handling location

source [18] notes HTTP API handles CORS at gateway level, while REST API with Lambda proxy requires Lambda to return CORS headers. **resolution:** sdk should support both patterns via configuration.

### anti-patterns

> "Using Access-Control-Allow-Origin: * allows any website to access your resources." — source [18] warns against wildcard CORS in production.

> "V2 imports the entire SDK (~70 MB)." — source [4] warns against using aws-sdk v2 due to bundle size impact on cold starts.
