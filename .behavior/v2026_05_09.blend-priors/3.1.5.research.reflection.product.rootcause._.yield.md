# reflection: root cause

> why is this needed?

---

## what is the symptom?

**observable problem:**
handler and client changes almost always propagate to each other. coordinate across two repos is friction.

**how it manifests:**
- change handler schema → must update client types separately
- add trail propagation → must implement in both repos
- upgrade dependency → must upgrade in both repos with separate PRs
- fix bug → may need to fix in both places

**when it occurs:**
- every time behavior crosses handler↔client boundary
- every dependency upgrade cycle
- every break change to contract

---

## 5 whys tree

### branch 1: coordination friction

1. **why is there coordination friction?**
   → changes must be made in two repos

2. **why must changes be made in two repos?**
   → handler and client are in separate packages (simple-lambda-handlers, simple-lambda-client)

3. **why are they in separate packages?**
   → they were built at different times for different purposes

4. **why were they built separately?**
   → client was needed to invoke handlers from other services; seemed like separate concern at the time

5. **why didn't we unify them then?**
   → **root cause: no unified vision for lambda SDK at the time; evolved organically rather than designed**

### branch 2: dated patterns persist

1. **why are the patterns dated?**
   → both repos use Joi, aws-sdk v2, middy v1

2. **why haven't they been updated?**
   → updates would require coordinated changes across both repos

3. **why is coordinated update hard?**
   → two release cycles, two test suites, two sets of consumers

4. **why does that matter?**
   → each update doubles the work; risk of version mismatch between repos

5. **why does version mismatch matter?**
   → **root cause: maintenance burden scales with repo count; friction compounds**

### branch 3: type safety gaps

1. **why are there type safety gaps between handler and client?**
   → schemas defined in handler repo, client repo infers types separately

2. **why can't client use handler schemas directly?**
   → different packages; would create circular dependency

3. **why would it be circular?**
   → handler depends on handler package, client depends on client package, neither can depend on both

4. **why can't both depend on shared types?**
   → no shared types package exists

5. **why doesn't shared types package exist?**
   → **root cause: architecture didn't anticipate tight couple; separate repos seemed sufficient**

---

## root cause summary

| layer | issue | is root cause? |
|-------|-------|----------------|
| surface | coordination friction between repos | no |
| structure | handler and client in separate packages | no |
| decision | built separately at different times | no |
| **architecture** | **no unified lambda SDK vision; evolved organically** | **yes** |
| **consequence** | **maintenance burden scales with repo count** | **yes** |

two related root causes converge:

1. **organic evolution without unified design** — repos grew separately instead of architected together
2. **repo count multiplies friction** — every cross-scope change costs N× effort for N repos

---

## recommendation

**address root cause: unify into single sdk-aws-lambda package**

this directly addresses both root causes:
- unified design: single package with coherent vision
- single repo: changes are atomic, no coordination

**what this enables:**
- shared types between handler and client
- single test suite for integrated behavior
- single release cycle
- atomic break changes

**what remains out of scope:**
- migrate all consumers immediately (will need deprecation period)
- support old dependency versions (clean break to Zod v4, aws-sdk v3, middy v7)

**follow-up needed:**
- migration guide for consumers of old packages
- deprecation notices on simple-lambda-handlers, simple-lambda-client
- timeline for old package end-of-life
