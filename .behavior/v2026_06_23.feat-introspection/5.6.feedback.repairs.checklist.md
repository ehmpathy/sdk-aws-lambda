# feedback repairs checklist

human review feedback on the introspection error + decomposition design, with the
repair required and why it matters. each item maps to a task (#id).

---

## [x] #12 — error classes must not override the HelpfulError constructor

**feedback:** the 4 lambda error classes each define a custom `constructor` that
hardcodes the message and reshapes metadata.

**why it matters:**
- helpful-errors subclasses are meant to be bare (`extends ConstraintError {}`) —
  same as `BadRequestError`, `UnexpectedCodePathError`, `ConstraintError`
  themselves. a custom constructor breaks the `new Error(message, metadata)`
  contract every other error in the codebase follows.
- a hardcoded message removes the caller's ability to state context at the throw
  site, where the real knowledge lives.
- consistency: a 3am-oncall reader expects every helpful-error to behave the same
  way (message first, metadata second, `.metadata`/`.code` accessors intact).

**repair:** make each class a bare `extends ConstraintError {}`; set message +
metadata at each throw site via `new LambdaX('clear message', { ... })`.

---

## [x] #13 — introspection-not-supported must state the reason

**feedback:** `LambdaIntrospectionNotSupportedError` is thrown for 3 different
causes with one opaque message; it never says *why*.

**why it matters:**
- "not supported" with no reason forces the reader to crack open the code to learn
  whether the handler threw, returned an empty payload, or returned a non-schema
  shape — three very different root causes.
- failloud: an error should carry enough context to diagnose without a debugger
  (rule.require.failloud).

**repair:** include a `reason` ('handler-error' | 'empty-payload' |
'non-schema-payload') in message + metadata, and carry through any
`functionError` detail.

---

## [x] #14 — getAllLambdaContracts must compose getOneLambdaContract

**feedback:** the two operations duplicate single-lambda introspection logic
separately; they should build upon each other.

**why it matters:**
- decomposition-for-recomposition: getAll is literally "getOne, for each function".
  a separate copy duplicates the introspection + parse + error logic and lets the
  two drift (e.g. one gates prep, the other detects blocked differently).
- one source of truth for "what introspection of one lambda means" keeps behavior
  coherent as the contract evolves.

**repair:** getAll = list-by-prefix → `getOneLambdaContract` per function →
assemble bare-name Record. all-or-none (see #17): a function that does not
support introspection aborts the whole batch; `LambdaIntrospectionBlockedError`
and `LambdaServiceNotFoundError` propagate too.

## [x] #15 — decompose getAllLambdaContracts dir by noun-prefix subdomains

**feedback:** the dir is too flat; subdomain relationships are hard to identify.

**why it matters:**
- a flat dir hides which leaves belong to which concern (function discovery vs
  contract parse vs record assembly), which raises read-time for humans and robots.
- noun-prefix groups (rule.prefer.most-common-denominator + bounded-contexts) make
  the structure self-evident and show shared leaves explicitly.

**repair:** group by subdomain; lift leaves shared with the peer
`getOneLambdaContract` (e.g. `asLambdaEndpointSchemaOrNull`) to a common ancestor
instead of cross-imports between peer dirs.

## [x] #16 — fix permadiff uuid in deployed.awsLambda snapshot

**feedback:** the deployed snapshot carries a fresh uuid every run.

**why it matters:**
- a permadiff snapshot is noise that trains reviewers to ignore snapshot changes —
  which defeats the whole point of snapshots (regression visibility).
- `--resnap` can never settle it, so ci/local diffs never go clean.

**repair:** pin the `exid` in the `[t2]` invalid-input case (like `[t0]`) or redact
`exid:*` before snapshot, then resnap.

## [x] #17 — getAllLambdaContracts must be all-or-none, not partial

**feedback:** why would a service ever have partial introspection support? a
function either has a contract or it does not — do not silently skip the ones
that lack it.

**why it matters:**
- a silently-skipped endpoint is a failhide: the caller asks for the service's
  contracts, gets a subset, and generates an sdk that omits endpoints — with no
  signal that a gap exists.
- all-or-none makes the invariant explicit: a service is introspectable as a whole,
  or the call fails loud and names the offender, so the gap is fixed at the source.

**repair:** getAll no longer catches `LambdaIntrospectionNotSupportedError`; any
unsupported function aborts the batch (the error names the function). Removed the
per-function skip path (`getOneContractEntryOrNull`). Replaces the earlier
"skip unsupported" decision in #14.

## [x] #18 — schema cast must be strict (no null), bare-name strip must be exact

**feedback:** why would the schema parser ever return null? it either has a
contract or does not support introspection. and bare-name extraction must
STRICTLY remove `{service}-{access}-`, not best-effort partial-match.

**why it matters:**
- `asLambdaEndpointSchemaOrNull` smuggled a third "maybe" state into a pure cast;
  a cast either yields the value or fails. null forced every caller to re-decide
  the same point and invited silent skips (#17).
- a loose prefix strip (slice-by-length + first-dash) can mis-key a Record when a
  name does not actually carry the expected prefix; exact-match-or-throw makes a
  malformed name fail loud instead of yield a wrong bare name.

**repair:**
- renamed `asLambdaEndpointSchemaOrNull` → `asLambdaEndpointSchema`; it now throws
  `LambdaIntrospectionNotSupportedError` (reason: non-schema-payload) instead of a
  null return.
- `asBareFunctionName` now takes `access`, requires the exact `{service}-{access}-`
  prefix, and throws `UnexpectedCodePathError` on any non-match (covered by a new
  unit test).
