# vision: introspection for lambda endpoints

## the outcome world

### day-in-the-life: after

a developer who wants to call `svc-invoice.getInvoice`:

```ts
// step 1: discover the contract for one endpoint
const contract = await getOneLambdaContract(
  { which: { service: 'svc-invoice', function: 'getInvoice' } },
  context,
);

// step 2: see the LambdaEndpointSchema
console.log(contract);
// LambdaEndpointSchema {
//   input: { type: 'object', properties: { invoiceId: { type: 'string' } }, required: ['invoiceId'] },
//   output: { type: 'object', properties: { id: { type: 'string' }, total: { type: 'number' }, ... } }
// }

// step 3: build caller with confidence
const invoice = await askLambdaEndpoint<GetInvoiceInput, Invoice>(
  { which: { service: 'svc-invoice', function: 'getInvoice' }, event: { invoiceId: 'inv_123' } },
  context,
);
```

a developer who wants to discover all endpoints in a service:

```ts
// get all contracts for a service
const contracts = await getAllLambdaContracts(
  { which: { service: 'svc-invoice' } },
  context,
);

// see all available endpoints: Record<string, LambdaEndpointSchema>
console.log(contracts);
// {
//   getInvoice: { input: {...}, output: {...} },
//   createInvoice: { input: {...}, output: {...} },
//   deleteInvoice: { input: {...}, output: {...} },
// }
```

### before/after contrast

| before | after |
|--------|-------|
| dig through source code to find schema | ask the endpoint directly |
| copy-paste types from handler repo | generate types from JSON schema |
| hope the types are up-to-date | always get current contract |
| deploy to discover errors | introspect in prep before deploy |
| no standard way to discover contracts | `getOneLambdaContract` / `getAllLambdaContracts` |

### the "aha" moment

> "i can just ask any endpoint what it expects and what it returns — no docs hunting, no source diving, no stale types."

## user experience

### usecases

1. **contract discovery** — caller wants to know what inputs an endpoint expects
2. **sdk generation** — team wants to auto-generate typed callers from JSON schema
3. **validation setup** — caller wants to validate their inputs before calling
4. **documentation** — team wants to expose endpoint contracts in internal docs

### contract inputs & outputs

**getOneLambdaContract input:**
```ts
{
  which: {
    service: string,   // e.g., 'svc-invoice'
    function: string,  // e.g., 'getInvoice'
  }
}
```

**getOneLambdaContract output:** `LambdaEndpointSchema`
```ts
interface LambdaEndpointSchema {
  input: JsonSchema;   // JSON schema for endpoint input
  output: JsonSchema;  // JSON schema for endpoint output
}
```

**getAllLambdaContracts input:**
```ts
{
  which: {
    service: string,   // e.g., 'svc-invoice'
  }
}
```

**getAllLambdaContracts output:** `Record<string, LambdaEndpointSchema>`
```ts
Record<string, LambdaEndpointSchema>
```

### what it looks like

```ts
// get one contract
const contract = await getOneLambdaContract(
  { which: { service: 'svc-invoice', function: 'getInvoice' } },
  context,
);
const inputSchema = contract.input;
const outputSchema = contract.output;

// get all contracts for a service
const contracts = await getAllLambdaContracts(
  { which: { service: 'svc-invoice' } },
  context,
);
const endpoints = Object.keys(contracts); // ['getInvoice', 'createInvoice', ...]
```

### timelines

1. **caller calls `getOneLambdaContract({ which })` or `getAllLambdaContracts({ which })`**
2. **function sends `{ introspect: 'schema' }` payload** (internal detail)
3. **handler middleware intercepts** (before user logic runs)
4. **middleware checks `env.access === 'prep'`** — if not prep, fails fast with clear error (introspection blocked in prod)
5. **middleware converts zod schemas to JSON schema**
6. **middleware returns schema response** (user logic never invoked)
7. **caller receives typed contract object**

## mental model

### how would users describe this to a friend?

> "you can call `getOneLambdaContract` to ask any lambda what it expects, or `getAllLambdaContracts` to discover all endpoints in a service — both return JSON schema."

### analogies

- **like GraphQL introspection** — query the API for its own schema
- **like OpenAPI /docs** — but for lambda-to-lambda calls
- **like `--help`** — ask the command what arguments it takes

### terms

| user term | our term | notes |
|-----------|----------|-------|
| "schema" | `contract.input`, `contract.output` | both input and output contracts |
| "get contract" | `getOneLambdaContract`, `getAllLambdaContracts` | caller-side functions |
| "introspect" | `{ introspect: 'schema' }` | internal trigger payload |
| "contract" | zod schema → JSON schema | machine-readable contract |
| "prep only" | `env.access === 'prep'` | security constraint |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| discover endpoint contract | ✅ yes | `getOneLambdaContract` or `getAllLambdaContracts` |
| generate SDK from schema | ✅ yes | JSON schema → TypeScript via tools |
| safe discovery | ✅ yes | prep-only prevents prod abuse |
| backward compatible | ✅ yes | extant callers unaffected |

### pros

- **zero config for consumers** — just call `getOneLambdaContract` or `getAllLambdaContracts`
- **always up-to-date** — schema comes from the handler itself
- **prep-only safety** — no introspection attack surface in prod
- **leverages extant zod schemas** — no new schema format to learn
- **JSON schema output** — industry standard, broad tool support

### cons

- **prep-only limitation** — can't introspect prod (intentional tradeoff)
- **schema size** — large schemas may have latency impact

### edgecases

| edgecase | pit of success |
|----------|----------------|
| introspect in prod | fails fast with clear error — no schema leak |
| ancient caller sends introspect | middleware still intercepts |
| malformed introspect payload | falls through to validation error |
| handler without output schema | returns null for output schema |

### alternatives considered

| alternative | why rejected |
|-------------|--------------|
| share types via npm package | not all callers are TypeScript; need cross-language support |
| publish schemas to registry | sync issues — registry schema may drift from deployed handler |
| generate types from source | requires build tool coordination; runtime introspection is simpler |

runtime introspection was chosen because it is always current, cross-language compatible, and requires no external coordination.

## groundwork

### external research

**zod v4 native JSON Schema:**
- zod v4 ships with native `z.toJSONSchema()` — no external dependency needed
- targets JSON Schema Draft 2020-12 by default (configurable to draft-07, draft-04, openapi-3.0)
- handles unions, discriminated unions, recursive schemas correctly
- **verified**: zod v4 native support ✅

### internal research

**extant contracts:**
- `genLambdaEndpoint` already requires `schema: { input: ZodSchema, output: ZodSchema }` (src/domain.operations/genLambdaEndpoint/genLambdaEndpoint.forAskEndpoint/genLambdaEndpoint.forAskEndpoint.ts:50-54)
- output validation already implemented via `genZodOutputValidationMiddleware`
- `@aws-sdk/client-lambda` already a dependency — `ListFunctionsCommand` available for `getAllLambdaContracts`

**extant vocab:**
- "contemp"/"ancient" for caller versions
- `_serde` for serialization metadata in responses
- `trail` for request correlation
- `askLambdaEndpoint` for caller-side invocation
- `getOneLambdaContract` — new term, parallels `askLambdaEndpoint` pattern
- `LambdaEndpointSchema` — new type, follows `LambdaEndpoint*` pattern
- `getAllLambdaContracts` — lists lambdas by service prefix, then calls `getOneLambdaContract` for each
- `introspect` — internal trigger, not exposed to callers

**extant patterns:**
- middleware chain: error handlers → io logger → trail → validation (src/domain.operations/genLambdaEndpoint/genLambdaEndpoint.forAskEndpoint/genLambdaEndpoint.forAskEndpoint.ts:113-119)
- early return pattern: `genConstraintErrorMiddleware` returns error response object

**env.access detection:**
- `askLambdaEndpoint` receives `context.env.access` (src/domain.operations/askLambdaEndpoint/askLambdaEndpoint.ts:44-45)
- handlers need similar access to env for introspection gate

## what is awkward?

### ~~env.access availability in handler~~ **resolved**

~~the handler currently receives only `ContextLogTrail` (log with trail). to gate introspection on `env.access = prep`, the middleware needs access to environment config.~~

**decision**: add `env` to `GenLambdaEndpointContext` — accepts static value OR async getter:
```ts
env:
  | { access: 'test' | 'prep' | 'prod' }
  | (() => Promise<{ access: 'test' | 'prep' | 'prod' }>)
```
mirrors `askLambdaEndpoint` context pattern; async getter enables lazy config fetch.

### introspect discriminator collision

what if a real endpoint has `{ introspect: string }` as a valid input field?

**context**: since `getOneLambdaContract` sends the introspect payload internally, this is an sdk implementation detail, not a caller concern.

**mitigations:**
- use less common key like `_introspect`
- middleware checks for exact match `{ introspect: 'schema' }`, not just presence of key

current lean: `{ introspect: 'schema' }` — risk is low since callers use `getOneLambdaContract`/`getAllLambdaContracts`, not raw payloads.

### middleware ordering

introspection middleware must run:
- **after** trail middleware (to have log context for debug)
- **before** input validation (introspect payload won't match input schema)

this requires inserting introspection middleware at the right position in the chain.
