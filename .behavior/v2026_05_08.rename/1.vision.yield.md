# vision: sdk-aws-lambda

## the outcome world

### day-in-the-life

a developer in a lambda service imports:

```ts
import { genLambdaEndpoint, askLambdaEndpoint } from 'sdk-aws-lambda';
import { genContextLogTrail } from 'sdk-logs';
```

when they define an endpoint, they get:
- input validation
- input/output log capture
- error classification (BadRequestError vs internal)
- automatic trace-id propagation via trail.exid

when they invoke another lambda, they get:
- typed responses
- error hydration
- automatic trail propagation (trace-id threads through)
- optional cache

trail auto-propagates through log context — trace-id correlates the entire request chain.

### before/after

| before | after |
|--------|-------|
| `npm install simple-lambda-handlers simple-lambda-client sdk-logs` | `npm install sdk-aws-lambda` |
| contract change requires coordination across two repos | single repo, single release |
| trace-id pass requires manual injection | auto-extracted from log context |
| three different release cycles | one release cycle |

### aha moment

"wait, i just pass `log` and the trace-id propagates automatically?"

yes. askLambdaEndpoint extracts trail from log.trail and injects it into the event. zero manual effort.

## user experience

### usecases

1. **define a lambda endpoint** - wrap business logic with validation, log trail, error classification
2. **ask a lambda endpoint** - call another service with typed request/response, trail auto-propagated
3. **correlate requests** - trace-ids thread through the call chain via log context

### contracts

**endpoints**:
```ts
// standard endpoint (direct invoke)
export const handler = genLambdaEndpoint.for.standard({
  log,
  schema,
  logic: async ({ event }) => { /* business logic */ },
});

// api gateway endpoint (http)
export const handler = genLambdaEndpoint.for.apiGateway({
  log,
  schema,
  logic: async ({ headers, body }) => ({
    statusCode: HTTPStatusCode.SUCCESS_200,
    body: { result },
  }),
  cors: { origins: ['example.com'] },
});

// sqs consumer endpoint
export const handler = genLambdaEndpoint.for.sqsConsumer({
  log,
  schema,
  logic: async ({ records }) => { /* process records */ },
});
```

**client**:
```ts
const result = await askLambdaEndpoint<ResponseType>({
  service: 'svc-jobs',
  stage,
  function: 'getJobByUuid',
  event: { uuid },
  log, // trail auto-extracted from log.trail and injected into event
});
```

**log context** (from sdk-logs):
```ts
import { genContextLogTrail } from 'sdk-logs';

const { log } = genContextLogTrail({
  trail: { exid: event.trail?.exid ?? null, stack: [] },
  env: { commit: process.env.COMMIT ?? null },
});
// log methods include trail.exid for correlation
```

### timeline

1. install sdk-aws-lambda + sdk-logs
2. import genLambdaEndpoint, askLambdaEndpoint, genContextLogTrail
3. define endpoints with schemas
4. ask other lambda endpoints
5. trace-ids thread automatically via log context

## mental model

### how users describe it

"it's like middy + aws sdk lambda invoke, but they talk to each other about trace ids"

"one package for both sides of lambda — define endpoints and get responses"

### analogies

- **http client/server in one**: like axios + express in one package, but for lambda
- **rpc framework**: similar to grpc where client and server share contract definitions

### vocabulary

| our term | user might say |
|----------|----------------|
| endpoint | lambda function, handler |
| askLambdaEndpoint | invoke, call |
| trail | trace, request-id, correlation-id |
| exid | trace-id, breadcrumbs |

## evaluation

### how well does it solve the goals?

| goal | solution |
|------|----------|
| contract changes propagate together | single repo, single release |
| trail.exid propagation | auto-extracted by askLambdaEndpoint from log context |
| modern best practices | fresh start, no legacy baggage |

### pros

- single dependency for lambda development
- coordinated releases (endpoint change + client change = one PR)
- trace-id propagation is automatic via log context
- consistent log format across endpoints and clients

### cons

- migration from simple-lambda-handlers + simple-lambda-client requires import updates
- larger package (combines two concerns)

### pit of success

- endpoints validate input by default (schema required)
- BadRequestError returns user errors without pollution of cloudwatch error metrics
- trace-ids propagate automatically when log context is passed

## open questions & assumptions

## what is awkward?

### scope split

this PR does only shell cleanup:
- prune sdk-logs references
- clear src/
- update readme + package.json to clean slate

the actual merge of simple-lambda-handlers + simple-lambda-client happens in a separate PR. this feels right — clean separation of concerns — but it means this PR produces a non-functional package momentarily.

### name uncertainty

sdk-aws-lambda is broad. it could encompass:
- endpoints (yes)
- client (yes)
- logs (via sdk-logs dependency)
- what else in the future?

the name is appropriately scoped for now, but may need refinement as scope becomes clearer.

### migration path

current users of simple-lambda-handlers and simple-lambda-client need a migration story. this vision doesn't address that yet — it's a concern for the merge PR, not this shell cleanup PR.
